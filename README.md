# vmbackup

Run the backup tooling directly from the git checkout. No script copying is required.

## Workflow

1. Clone the repo where you want it to live on the hypervisor.
2. Run `sudo ./setup.sh`.
3. Answer the prompts for backup paths, restore image directory, remote sync, alerts, and cron.
4. For updates later, `git pull` in the same checkout and rerun `sudo ./setup.sh` only if you need to change config or cron.

## Backup Model

`vm_backup.sh` uses libvirt external snapshots instead of `virtnbdbackup`.

For each running VM it will:

1. save the domain XML and disk manifest
2. create an external disk-only snapshot, with guest quiesce when available
3. stream each stable base disk through `zstd` directly to the remote host when `REMOTE_HOST` is configured
4. pivot the VM back to the base image with `blockcommit`

If no remote host is configured, it stores local `.zst` full backups under `LOCAL_BACKUP_PATH`.

This avoids QEMU NBD socket handling and avoids large local staging copies when backing up remotely.

## Scripts

- `setup.sh`: interactive setup for `config.sh`, log files, and cron
- `test_notifications.sh`: sends ntfy and email test notifications using the same alert format as the backup script
- `vm_backup.sh`: snapshot-stream backup runner
- `vm_restore.sh`: restore the newest `full-*` backup for a VM from remote or local compressed backups
- `vm_retention.sh`: prune old `full-*` backup directories
- `cleanup.sh`: remove cron, logs, temp data, and backups while leaving the repo checkout in place

## Notes

- `config.sh` is generated by `setup.sh` and should stay out of git.
- `setup.sh` accepts either a full ntfy URL or a bare topic and normalizes bare topics to `https://ntfy.sh/<topic>`.
- email alerts require explicit `from` and `to` addresses plus a Resend API key
- `setup.sh` writes cron entries that point back to this checkout, so `git pull` updates what cron runs.
- `zstd` must be installed on the hypervisor.
